No reciprocal star, follow, or promotional request is made from these signals.
The only useful next step for someone who arrived through a star, fork, or
profile visit is optional maintainer evidence:

- open trial feedback with a repository or workflow link
- run the no-fail workflow in a maintainer-owned public repository
- reply with a concrete reason the report is not useful
- open a focused PR only when there is a real repository-specific fix

Do not tag stargazers or fork owners just to create activity. If a fork or star
turns into a public run, maintainer reply, focused PR, or trial-feedback issue,
then link that new public artifact here and in the evidence ledger.

## Recognition Policy

The project recognizes public work that helps maintainers evaluate or improve
the tool:

- merged contributor PRs
- maintainer replies with concrete feedback
- public workflow runs in maintainer-owned repositories
- accepted external PRs informed by an `oss-signal` audit
- neutral or negative trial feedback that improves the rules

Recognition means linking the public contribution in the evidence ledger,
reviewer docs, [ACKNOWLEDGEMENTS.md](../ACKNOWLEDGEMENTS.md), release notes
when relevant, or future thank-you comments when write access is available.
For social replies, use the bounded templates in [social-launch.md](social-launch.md)
only when the thread is directly relevant.

## Reciprocity Boundary

`oss-signal` does not do star-for-star, follow-for-follow, reciprocal PRs, or
repeated comment bumps. A star can be appreciated, but it is not evidence of
adoption and should not be repaid just to influence a reviewer.

It is fine for maintainers to star repositories they independently find useful.
Those stars are not tracked here as application evidence.
